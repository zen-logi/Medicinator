# コメント規約

コメントは意図、制約、読み取りにくい挙動を説明するために使います。コードを読めば分かる内容を繰り返さないでください。

## コメントを書く場面

- 業務ルールが文脈なしでは誤読されやすい
- セキュリティ、プライバシー、デプロイ上の制約が実装に影響する
- プラットフォーム、フレームワーク、外部 API の制限により回避策がある
- テストフィクスチャが特定のエッジケースを表している

## コメントを避ける場面

- コード自体で意図が十分に分かる
- コメントがメソッド名、型名、代入内容を言い換えているだけ
- 実装変更後に古い挙動を説明したまま残っている

## ドキュメントの書き方

- リポジトリ内ドキュメントは簡潔な Markdown で書く
- 抽象的な説明より、具体的なコマンドとファイルパスを優先する
- 公開ドキュメントにシークレットや非公開の運用詳細を含めない
- ローカルの一時作業場所として `D:\tmp` を明記する

## C# XML コメント

- `summary` は日本語で書く
- 「ですます」調を使わない
- 末尾に句点を付けない
- Service 実装クラスの public method は `inheritdoc` を使う
- private method は `summary` を書く
